# Taste Layer Backlog

Site-O-Mattic should stay block-first: core WordPress blocks and global settings carry the design, while generated CSS is reserved for shared polish, accessibility, and responsive edge cases. The taste layer is Codex vision and reasoning from rendered screenshots; scripts, scores, dashboards, and validators are evidence and regression checks only.

## Active Review Lanes

- Visual composition: first viewport balance, proof timing, logo/header weight, repeated formulas, and screenshot evidence.
- Typography: type scale, measure, role contrast, mobile wrapping, button labels, and avoiding CSS rescue typography.
- Color theory: value contrast, warm/cool balance, action color discipline, and palette fingerprints that reflect rendered behavior.
- Lean output: custom CSS size, `!important`, negative margins, transforms, sticky/fixed UI, wildcard selectors, and fragile breakpoints.
- Block-first styling: visual decisions should live in block styles, block settings/supports, `theme.json`-compatible global styles, and block attributes before scoped CSS. CSS that duplicates what core blocks can express is design debt.
- Catalog presentation: GitHub Pages and dashboards should help judge rendered design quality quickly, not just list files.

These commands do not approve visual quality. A pass is only ready to describe as tasteful after Codex inspects the desktop/mobile screenshots, explains the visual change in human design language, and, for non-trivial changes, gets an independent model/subagent taste read.

## Highest-Priority Taste Work

1. Reduce CSS-rescue design.
   The new `blueprint:lean-css` report shows heavy generated CSS across the catalog, especially `photo-booth-rental`, `closet-pantry-organization`, `small-event-dj-sound`, `micro-wedding-florals`, and the menu/gallery variants. Next refactor target: replace transforms, negative margins, sticky/fixed UI, and broad wildcard proof-card selectors with block attributes, block settings/supports, `theme.json`-compatible global styles, and simpler layout choices. Treat code metrics as guardrails; decide taste by rendered screenshot critique.

2. Tighten color-role variety.
   Global styles, shared CSS variables, and generated button markup now resolve `colorRoles`, and the uniqueness report has a blocking color-role fingerprint budget. The top color-role cluster is now 12/35, down from 16/35 and inside the 35% target. Next step: keep color canaries paired with rendered contrast checks instead of broad palette churn.

3. Add rendered color checks beyond H1.
   Visual sweep now catches first-viewport H1 contrast, but buttons, nav, proof cards, chips, muted labels, and text-over-image cases still need rendered checks. Add a `blueprint:color` report or extend `visual:sweep` with sampled checks for those roles.

4. Move typography back into block presets.
   Several builders still emit inline heading styles, but the generator now normalizes headings, buttons, nav, uppercase labels, and utility labels to token-safe block attributes/inline styles before the Blueprint is written. Keep moving hero/card builders toward direct preset output so the normalizer becomes a safety net instead of the main typography author. `visual:sweep` now emits taste warnings for wrapped short CTA labels, rendered H1 line count/height, and navigation overflow/wrapping across desktop, mobile, and tablet viewports.

5. Break visual nearest-neighbor clusters.
   Current visual sweep nearest neighbors are prompts for human/model review, not automatic redesign orders. The latest full sweep has 35/35 rendered, 0 failures, and 0 taste warnings; the closest pairs now start with balloon/headshot at distance 50. Treat close pairs as taste bugs only after screenshot inspection confirms the pages actually feel too similar. Typography warnings are resolved; the next taste work is silhouette separation and pattern reassignment when the rendered pages fail the squint test, not palette-only churn.

6. Improve mobile proof timing.
   `visual:sweep` records `mediaProof` and emits `tasteWarnings` when less than 25% of the intended proof media is visible on mobile first viewports. The latest full-catalog sweep passed with 35 total, 0 failures, and 0 taste warnings. Keep this evidence active, and prefer block-order fixes, shorter hero flows, and smaller core spacing before adding sticky action CSS.

7. Keep the catalog a review surface.
   `docs/index.html` now features real desktop/mobile rendered screenshots for the highlighted blueprint and compact link cards for the rest. Each card includes compact review-signal chips from visual sweep evidence: sweep status, CTA, silhouette, nearest neighbor, and taste-warning count when present. Keep future catalog improvements text-first unless a screenshot directly improves design review.

## Calibration Notes

The lean CSS report currently counts shared CSS and variant CSS together, so its scores are directional rather than release-failing. The transform parser now avoids `text-transform:` false positives; the next calibration pass should split shared baseline weight from per-variant additions and fail only on avoidable variant debt.

Taste warnings are also directional while thresholds are calibrated. They should appear in `report.json`, `review-evidence.json`, and the visual comparison dashboard as prompts for design review, not as release blockers until their false-positive rate is known. Current warning families include mobile media proof, short CTA wrapping, H1 wrapping/height, nav overflow/wrapping, and too-close nearest-neighbor pairs.
